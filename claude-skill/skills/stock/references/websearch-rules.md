# WebSearch 의무 룰

> 로드 시점: BLOCKING 체크 #4·#7 + 2단계 데이터 수집 + 재실행 시.

## 종목별 WebSearch 필수 룰 (BLOCKING #7 보강)

각 보유 종목마다 **최소 1회** WebSearch 호출. 쿼리 포맷 표준화:

```
"{종목명} YYYY-MM-DD 공시 뉴스"
"{종목명} 실적 공시 YYYY-MM"  (실적 시즌이면 추가)
"{종목명} 실적 발표일"  (D-7 이내 시즌이면 필수)
```

## 특정 조건 시 추가 WebSearch 필수 (위 기본 1회 외)

| 조건 | 추가 쿼리 |
|---|---|
| 일일 등락률 **±3% 이상** | "{종목명} 급등 or 급락 원인 YYYY-MM-DD" |
| 거래량 20일 평균 대비 **1.5배+** | "{종목명} 거래량 증가 공시" |
| **52주 신고가/신저가** 갱신 | "{종목명} 신고가 원인" |
| **실적 발표 D-7 이내** | "{종목명} 실적 예고 컨센서스" |
| 이상 수급 (z-score ±2+) | "{종목명} 기관 외국인 동향" |

## 결과 활용

- daily 보고서의 `## 뉴스 / 촉매` 섹션에 기록 (없으면 "해당 없음")
- 유의미한 팩트는 `## 📌 Base 영향도 판단` 섹션으로도 승격
- **"원인 모름" 채 액션 플랜 잡기 금지** — ±3%+ 움직임 있는데 뉴스 안 찾으면 Dependency Audit에 ❌ 표시

## 재분석/재실행 시 WebSearch 의무 (v12.2 보강)

**동일 날짜에 daily 재실행 / 부분 갱신 / 단건 종목 재분석**할 때도 **WebSearch 캐시 재사용 금지**.

### 이유

- 시간이 지나면 새 공시·뉴스가 추가됨 (장중 → 장 마감 후 → 시간외)
- 실적 발표일 같은 핵심 이벤트는 아침에 미발생 → 오후에 발표 케이스 빈번
- 가격 변동(NXT 시간외 등)이 새 뉴스 결과인 경우 다수

### 재실행 시 추가 룰

| 트리거 | 행동 |
|---|---|
| 같은 날 두 번째 분석 | 종목별 WebSearch **재실행 의무** (시간 차이 ≥ 30분이면 100%) |
| 가격이 직전 분석 대비 ±2% 변동 | 해당 종목 WebSearch **추가 1회** ("종목명 시간외 뉴스" 등) |
| 실적 시즌 종목 (D-7 이내) | **매 분석마다 필수** — 시즌 동안 캐시 재사용 절대 금지 |
| NXT/시간외 가격 vs KRX 종가 차이 ±3%+ | 해당 종목 추가 WebSearch ("종목명 장후 시간외") |

### 캐시 재사용 허용 범위

- 가격·지표 데이터 (`compute_indicators`, `compute_signals` 결과) — 같은 분(分) 내라면 재사용 가능
- 재무 데이터 (`compute_financials`) — 분기 단위 갱신이라 동일 세션 내 재사용 OK
- WebSearch 뉴스 결과 — **재사용 금지** (위 룰 적용)

### Dependency Audit 갱신

- 재분석 시에도 "WebSearch 종목별 N/N" 항목을 **새로** 카운트 (직전 분석 결과 인용 금지)
- 캐시 재사용으로 스킵 시 ❌ 표시 + 사유 명시

## compute_signals 서버 에러 시 우회

- 지표(`compute_indicators`) + 이벤트(`detect_events`) + 등급(`compute_score`)을 조합해 **수동 verdict 산정**
- daily 보고서에 "⚠️ compute_signals 실패 — 지표 기반 수동 판정" 명시
